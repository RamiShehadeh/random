Recording flow: Core.Publish<T> increments a depth counter (2 lines added to your existing code), then fans out to all subscribers as normal. The MessageRecorderService is just another subscriber — but its handler checks PublishDepth <= 1 and _isRecording before doing anything. When recording, it writes the raw struct bytes into a flat MessageTape via pointer cast. No serialization, no boxing.
Playback flow: The MessagePlaybackService iterates the tape’s entry index, looks up a pre-built Action<Core, MessageTape, int> by typeId (array index, not dictionary), and that action reads the raw bytes back as T and calls Core.Publish(in message). Your existing Services and Views receive the message exactly as they would have originally.
Why This Specific Design
The universal subscriber uses reflection only once at startup — RegisterAllMessageTypes scans assemblies and calls RegisterTyped<T> per type via MakeGenericMethod. After that, each handler is a closed generic delegate with zero reflection. The per-message hot path is: one bool check → one int comparison → one sizeof(T) memcpy.
PublishDepth is checked inside the subscriber callback, not in Core.Publish itself. This means Core only needs 2 added lines (increment/decrement) and doesn’t need to know the recorder exists. The depth check <= 1 (not == 0) is because by the time the subscriber callback fires, Core has already incremented depth to 1 for the current publish.
Three playback modes because they serve different purposes: Instant dumps everything in one frame for state reconstruction, FrameSynced replays at the original frame cadence for behavior testing, and Timed uses wall-clock timestamps for real-time replay demos
